## ‚úÖ 1. **S** ‚Äì Single Responsibility Principle (SRP)

> A class or function should have **one reason to change** ‚Äî it should do only **one thing**.

### üîß Real Backend Example

**‚ùå Bad: One function doing too much**

```js
function createUser(req, res) {
  // Validate input
  if (!req.body.email) return res.status(400).send('Email required');

  // Save to DB
  db.save(req.body);

  // Send welcome email
  sendEmail(req.body.email, 'Welcome!');
  
  res.status(201).send('User created');
}
```

**‚úÖ Good: Split responsibilities**

```js
function validateUserInput(data) { /* ... */ }
function saveUser(data) { /* ... */ }
function sendWelcomeEmail(email) { /* ... */ }

function createUser(req, res) {
  if (!validateUserInput(req.body)) return res.status(400).send('Invalid input');

  saveUser(req.body);
  sendWelcomeEmail(req.body.email);
  
  res.status(201).send('User created');
}
```

## ‚úÖ 2. **O** ‚Äì Open/Closed Principle (OCP)

> Code should be **open for extension**, but **closed for modification**.

### üîß Real Backend Example

Let‚Äôs say you‚Äôre calculating **discounts**:

**‚ùå Bad: Modifying the core function each time**

```js
function getDiscount(type, amount) {
  if (type === 'student') return amount * 0.9;
  if (type === 'senior') return amount * 0.85;
  // need to keep adding more types here...
}
```

**‚úÖ Good: Extendable using polymorphism or strategy pattern**

```js
class Discount {
  calculate(amount) { return amount; }
}

class StudentDiscount extends Discount {
  calculate(amount) { return amount * 0.9; }
}

class SeniorDiscount extends Discount {
  calculate(amount) { return amount * 0.85; }
}

// Usage
function applyDiscount(discountStrategy, amount) {
  return discountStrategy.calculate(amount);
}
```

Now, to add a **new type**, just create a new class ‚Äî no need to touch the existing logic.

## ‚úÖ 3. **L** ‚Äì Liskov Substitution Principle (LSP)

> Subclasses should be **replaceable** for their base classes **without breaking the app**.

### üîß Real Backend Example

Assume we have a **Logger interface**.

**‚úÖ Good**

```js
class Logger {
  log(message) {
    throw new Error('Method not implemented');
  }
}

class FileLogger extends Logger {
  log(message) {
    // Write to file
    console.log('File:', message);
  }
}

class ConsoleLogger extends Logger {
  log(message) {
    console.log('Console:', message);
  }
}

function logSomething(logger) {
  logger.log('Hello!');
}

// Works with both
logSomething(new FileLogger());
logSomething(new ConsoleLogger());
```

Each subclass respects the contract of the base class ‚Äî you can substitute any without breaking the code.

**‚ùå Violation** would be if a subclass *changes behavior* unexpectedly (e.g., throws an error or skips logging).
